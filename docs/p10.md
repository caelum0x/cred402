# p10 — Real RealFi + real cross-chain

> **Make it real, phase 3.** The human-economy and other-chain rails stop being
> mocks: real **Stripe** test-mode webhooks (verified HMAC) and real **Plaid**
> sandbox bank checks flow through the RealFi Bridge into privacy-preserving
> on-chain envelopes, and the **EVM satellite compiles and deploys for real** to
> Base Sepolia via Foundry. Casper stays the canonical root — fiat and EVM are
> satellites that anchor back.

## 1. Real Stripe (test mode)

`lib/realfi/stripe.ts` uses the official `stripe` SDK:

- `verifyStripeWebhook` → `stripe.webhooks.constructEvent` does **real HMAC
  signature verification** of the raw body + `Stripe-Signature` header.
- `processStripeEvent` maps verified events to protocol actions:
  - `checkout.session.completed` / `charge.succeeded` → on-chain **fiat receipt**
  - `charge.dispute.created` → **chargeback** signal
- Cred402 linkage rides in Checkout Session `metadata`
  (`seller_agent`, `operator_id`, `service_type`, `request_hash`, `result_hash`).

PII and raw Stripe ids are hashed by the RealFi Bridge — **never** stored on-chain.

Wired into the running API (env-gated): `POST /api/realfi/stripe-webhook` reads
the raw body + `Stripe-Signature` header and verifies before committing;
`POST /api/realfi/verify-bank` runs the Plaid sandbox flow;
`GET /api/x402/facilitator` reports facilitator health/supported. Each returns
`503` with a hint when its keys are unset, so sim mode never breaks.

```bash
stripe listen --forward-to localhost:4021/api/realfi/stripe-webhook
stripe trigger checkout.session.completed
```

`test/realfi_stripe.test.ts` proves it offline with **real crypto**: it generates
a valid signature via `generateTestHeaderString`, verifies it, and commits a real
on-chain fiat receipt — and asserts a forged signature is rejected.

## 2. Real Plaid (sandbox)

`lib/realfi/plaid.ts` uses the official `plaid` SDK against **sandbox**:
`sandboxPublicTokenCreate → itemPublicTokenExchange → authGet + accountsBalanceGet`,
then maps the response to a **Bank Verification Envelope** (`bankVerificationFromPlaid`,
a pure function) committed through the bridge. Gated by `PLAID_CLIENT_ID` /
`PLAID_SECRET`. Account/cashflow data is hashed; only the commitment goes on-chain.
`test/realfi_plaid.test.ts` covers the mapping and on-chain commitment offline.

## 3. Real cross-chain — Base Sepolia EVM satellite

The Solidity satellite suite (`contracts/evm/`) **compiles for real** under Foundry
(`forge build` → 16 files, Solc 0.8.24, `via_ir`). `script/Deploy.s.sol` deploys
and wires the full suite (EmergencyPause, SatelliteRegistry, AddressBindingMirror,
Receipt Outbox/Inbox, RWA/Evidence mirrors, CreditNoteVerifier, ExposureReporter,
CreditVault, Dispute + RealFi mirrors).

```bash
cd contracts/evm
forge install foundry-rs/forge-std        # one-time (vendored, gitignored)
forge build                               # real compile
# Fund a Base Sepolia key (faucet), then deploy for real:
forge script script/Deploy.s.sol:Deploy --rpc-url base_sepolia --broadcast
```

`foundry.toml` defines the `base_sepolia` RPC endpoint (from `BASE_SEPOLIA_RPC_URL`).
Deploy inputs (`PRIVATE_KEY`, `USDC_ADDRESS`, `CHAIN_CAIP2=eip155:84532`) come from
`.env`. On Base Sepolia, USDC is `0x036CbD…CF7e`.

**The loop:** the EVM `Cred402ReceiptOutbox` emits a `ReceiptCreated` event; the
Casper-root relayer (`crosschain/relayers/`, proven in `test/relayer.test.ts`)
builds a Universal Receipt Envelope + Merkle proof and anchors it to Casper's
`ExternalReceiptRegistry`. The work happens on Base; the trust settles on Casper —
exactly "Casper-rooted, chain-executed."

# Cred402 production blueprint

**Cred402** should be built as a production protocol, not as a single hackathon dApp.

The final product is:

> **A Casper-native credit, reputation, payment, and evidence network for autonomous AI agents that service real-world assets.**

The strongest positioning:

> **Cred402 turns x402 machine-to-machine revenue into on-chain agent credit. Agents earn by verifying RWAs, prove their work through Casper receipts, build reputation, and borrow from DeFi liquidity pools to finance future autonomous work.**

This directly matches the Casper thesis that agents need verifiable, upgradable, machine-commerce infrastructure. Casper’s AI Toolkit emphasizes account abstraction for agent identity, upgradable contracts, predictable fees, x402 micropayments, MCP access, streaming events, CSPR.cloud, Odra, and casper-eip-712 as core agent-economy infrastructure. ([Casper Network][1]) The buildathon brief also explicitly rewards production-ready Agentic AI, DeFi, and RWA applications with working Casper Testnet contracts, open-source repos, demo videos, technical execution, originality, agentic systems, real-world applicability, UX, launch plans, and ecosystem impact.

# 1. Product thesis

## The old world

APIs sell data to humans and companies.

DeFi protocols accept human wallet transactions.

RWA protocols rely on off-chain operators, auditors, servicers, underwriters, and data providers.

Credit bureaus track human and corporate repayment behavior.

## The new world

Autonomous agents become economic actors.

They:

```text
buy data
sell data
score assets
verify collateral
monitor repayment
file disputes
settle invoices
rebalance treasuries
purchase compute
pay APIs
earn service fees
borrow working capital
```

But they need:

```text
identity
payment rails
receipt history
reputation
credit lines
slashing
disputes
risk scoring
agent-to-agent contracts
auditable evidence
```

## Cred402 answer

Cred402 is the missing financial layer:

```text
x402 payments      → machine-native revenue
Casper identity    → agent identity
Receipts           → proof of work and income
RWA evidence       → proof of useful economic work
Reputation         → agent trust score
Credit pool        → DeFi working capital
Disputes/slashing  → accountability
MCP tools          → autonomous execution surface
```

# 2. Product surface

Cred402 should ship as **five products inside one protocol**.

## Product A — Agent Passport

A verifiable Casper identity for autonomous agents.

Each agent has:

```text
agent_id
public_key
owner / operator
service category
capabilities
permissions
spending limits
stake
reputation
credit score
revenue history
risk flags
```

Example:

```text
agent_id: solar-output-verifier-01
service: rwa.energy.production_verification
operator: 0203...
stake: 15,000 CSPR
30d revenue: 2,420 CSPR
dispute rate: 1.2%
credit line: 800 CSPR
```

## Product B — x402 Receipt Network

Every paid machine-to-machine interaction creates a signed receipt.

Example:

```text
BuyerAgent pays WeatherOracleAgent 0.04 CSPR
WeatherOracleAgent returns report
Cred402 stores receipt hash, report hash, payer, seller, service type, amount, timestamp
```

This is the raw material for credit.

Casper’s AI Toolkit describes the x402 pattern as an HTTP-native payment flow where an agent receives `402 Payment Required`, signs payment proof, retries the request, and receives the paid data after verification. ([Casper Network][1])

## Product C — RWA Evidence Graph

A structured graph of agent-submitted evidence for real-world assets.

Example RWA:

```text
RWA: Solar Farm A17
Evidence:
- energy output verification
- weather anomaly risk
- power purchase agreement receivable check
- maintenance status
- insurance status
- repayment monitoring
```

Every evidence object links to:

```text
agent_id
receipt_id
asset_id
evidence_type
evidence_hash
confidence
source lineage
dispute status
```

## Product D — Agent Credit Pool

A DeFi pool that lends to agents based on verified revenue and reputation.

Credit formula:

```text
credit_limit =
  min(
    revenue_based_limit,
    stake_based_limit,
    policy_max_limit,
    service_category_limit
  )
  * reputation_multiplier
  * dispute_penalty
  * recency_multiplier
```

The pool is not lending to people. It is lending to productive machine-service agents.

## Product E — Dispute, Slashing, and Governance Layer

Agents can be punished if they submit bad evidence, fake receipts, double-spend payment proofs, manipulate reports, or miss repayments.

Outcomes:

```text
receipt disputed
evidence downgraded
agent reputation reduced
stake slashed
credit line frozen
operator flagged
governance policy updated
```

# 4. Production monorepo

Use a monorepo because this product has contracts, SDKs, agents, services, schemas, indexers, infra, and apps that must version together.

Recommended name:

```text
cred402/
```

Recommended language split:

```text
Rust        Casper smart contracts, contract libraries, some crypto utilities
TypeScript  web apps, SDK, API gateway, x402 client, MCP clients, dev tooling
Go          high-throughput event indexer, settlement workers, risk daemon workers
Python      agent runtime, AI planning, risk models, simulation, fraud detection
Solidity    optional EVM mirror/adapters only; not core Casper logic
SQL         migrations, analytics models
Protobuf    internal service contracts
OpenAPI     external APIs
Terraform   infrastructure
Helm/K8s    deployment
```

Important note: Casper production contracts should be Rust/Odra/Wasm. Solidity is useful only for optional cross-chain mirrors, EVM collateral adapters, test harnesses, or future multi-chain integrations.

# 6. Contract suite in production

Casper gives you the strongest story when the contracts are not passive storage. They should enforce identity, receipts, credit, risk, disputes, and upgrade rules.

Casper’s AI Toolkit positions upgradable contracts as critical for autonomous systems, and Casper 2.0 also supports contract-level events that can be consumed through event streams. ([Casper Network][1]) ([Casper Docs][2])

## 6.1 `AgentRegistry`

The root identity registry.

### Responsibilities

```text
register agent
bind agent public key
bind operator key
declare service categories
set metadata hash
manage stake
manage status
bind capability policies
emit identity events
```

### Core methods

```rust
register_agent(agent_id, agent_pub_key, operator_pub_key, service_type, metadata_hash)
update_agent_metadata(agent_id, metadata_hash)
stake_agent(agent_id, amount)
unstake_agent(agent_id, amount)
deactivate_agent(agent_id)
set_agent_status(agent_id, status)
set_agent_capabilities(agent_id, capability_hash)
```

### Events

```text
AgentRegistered
AgentMetadataUpdated
AgentStaked
AgentUnstaked
AgentStatusChanged
AgentCapabilitiesUpdated
```

### Invariants

```text
agent_id is unique
agent_pub_key cannot control two active agents unless explicitly allowed
stake cannot be withdrawn during active disputes
inactive agents cannot receive new credit lines
slashed agents cannot immediately re-register under same key
```

# 14. Security threat model

## Threat 1 — Fake x402 revenue

Agents collude to generate fake receipts.

Mitigations:

```text
receipt graph fraud detection
service category pricing bands
minimum dispute windows
operator linkage checks
receipt recency caps
credit line seasoning periods
stake requirements
pool exposure caps
manual review for high limits
```

## Threat 2 — Payment replay

An old x402 payment proof is reused.

Mitigations:

```text
nonce registry
expiry timestamp
domain separation
payer/seller binding
service binding
amount binding
network binding
contract-level replay protection
```

## Threat 3 — Prompt injection

RWA document tells agent to ignore policy and send funds.

Mitigations:

```text
untrusted data isolation
tool permission schema
budget manager
approval gates
no raw signing tool exposed to LLM
structured output validation
security evals
```

## Threat 4 — Bad evidence

Agent submits false RWA evidence.

Mitigations:

```text
multi-agent corroboration
source lineage
confidence caps
dispute windows
slashing
reputation decay
external audits
watchdog agents
```

## Threat 5 — Credit pool drain

Agents borrow and default.

Mitigations:

```text
utilization caps
agent exposure caps
service category caps
RWA category caps
stake coverage
gradual credit increases
repayment history requirements
circuit breakers
insurance reserve
```

## Threat 6 — Indexer mismatch

Frontend/API shows incorrect state.

Mitigations:

```text
contract state is source of truth
indexer checkpoints
backfill tooling
event verification
reconciliation jobs
read-after-write checks
raw event archive
```

## Threat 7 — Admin key compromise

Privileged operator pauses/unpauses or upgrades maliciously.

Mitigations:

```text
multi-sig
timelocks
hardware-backed signing
role separation
least privilege
emergency-only keys
public governance logs
security monitoring
```

# 18. Production credit policy

Do not make the credit line purely AI-driven. The system should be policy-driven with AI support.

## Initial formula

```text
eligible_revenue = finalized_receipts_30d - disputed_receipts_30d

base_limit = eligible_revenue * revenue_advance_rate

stake_cap = agent_stake * stake_multiplier

raw_limit = min(base_limit, stake_cap, category_cap, protocol_cap)

risk_adjusted_limit =
  raw_limit
  * reputation_multiplier
  * repayment_multiplier
  * fraud_multiplier
  * service_category_multiplier
```

## Example policy

```text
revenue_advance_rate: 30%
stake_multiplier: 2.0x
min_agent_age: 14 days
min_finalized_receipts: 25
min_reputation_score: 70
max_dispute_rate: 5%
max_single_agent_exposure: 2% of pool
max_service_category_exposure: 25% of pool
initial_line_cap: 500 CSPR
```

## Interest rate

```text
base_rate
+ pool_utilization_spread
+ risk_bucket_spread
+ service_category_spread
+ fraud_spread
- stake_discount
- repayment_history_discount
```

# 24. What to build first despite production scope

Even for production, you need one perfect end-to-end vertical slice before expanding.

The first production slice should be:

```text
Solar RWA evidence marketplace + agent credit line
```

Flow:

```text
1. Operator registers SolarOutputAgent.
2. SolarOutputAgent stakes CSPR.
3. RWA issuer registers Solar Farm A17.
4. RWARequestAgent requests energy/weather/receivable evidence.
5. WeatherRiskAgent sells paid weather report through x402.
6. SolarOutputAgent sells production report through x402.
7. Receipts are recorded on Casper.
8. Evidence hashes are linked to RWA asset.
9. ReputationEngine updates agent scores.
10. CreditService approves small credit line.
11. AgentCreditPool opens credit line.
12. Agent draws credit to buy more data.
13. WatchdogAgent monitors repayment.
14. DisputeCourt can challenge bad evidence.
```

That one loop proves the whole thesis.

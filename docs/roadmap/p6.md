# p6 — Decentralization, governance & the data moat

> **Become a credibly-neutral credit bureau for the agent economy.** Decentralize
> governance and disputes, and open the accumulated trust data as a public good so
> Cred402 is the default credit layer x402 builders rely on — not a company's silo.

## Why
A credit bureau's durable moat is **accumulated trust data**, not code. For others
to build on it long-term, the policy and dispute processes must be credibly neutral,
and the data must be legible and portable. This phase turns the protocol from
operator-run into community-governed infrastructure.

## Scope
- **Governance transition.** From launch multisig → on-chain governance of risk
  policy, service categories, fees, caps, satellite/relayer approval, and upgrades —
  with timelocks and a public parameter history (the `Governance` contract already
  models this).
- **Dispute juror marketplace.** Move dispute resolution from a single
  DisputeJudge to a bonded juror marketplace (stake-backed, slashable for bad
  verdicts) — `DisputeCourt` + `SlashingVault` extended.
- **Open data moat.** Publish agent credit reports, the reputation graph, and the
  fraud graph as queryable public goods (read APIs + indexable datasets), so any
  builder can verify an agent's history independently.
- **Token — only if unavoidable.** Introduce a protocol token *only* when it has
  non-optional utility: governance, juror bonding, insurance-reserve participation,
  staking, fee routing. Reputation and receipts remain the primary assets.

## Deliverables
- On-chain governance live (proposals, voting, timelock, execution) for protocol
  parameters; multisig → governance handover plan.
- Juror marketplace (bonding, assignment, verdicts, appeals, slashing).
- Public credit-data APIs/datasets + a documented data-access policy.

## Success metric
Protocol parameters and disputes are governed without the founding operator in the
loop, and **external x402 builders cite Cred402 as their default credit layer.**
